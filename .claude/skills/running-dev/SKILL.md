# 러닝크루 앱 개발 오케스트레이터

6인 에이전트 팀(planner, domain-analyst, backend-dev, flutter-dev, test-engineer, qa)을 조율해 기능을 개발한다. 에이전트 정의는 `.claude/agents/`, 도메인 규범은 `domain-model` 스킬 참조.

**실행 모드: 에이전트 팀 (기본).** 팀 도구(TeamCreate/SendMessage/TaskCreate)가 사용 가능하면 팀을 구성해 자체 조율시킨다. 팀 도구가 없는 환경이면 서브 에이전트 모드로 폴백한다(아래 폴백 절 참조). 모든 Agent 호출에 `model: "opus"`를 명시한다.

## Phase 0: 컨텍스트 확인 (매 실행 필수)

시작 전에 실행 유형을 판별한다:

| 상황 | 판별 | 실행 모드 |
|------|------|----------|
| `_workspace/` 없음 | 초기 실행 | 전체 파이프라인 |
| `_workspace/` 있음 + 부분 수정 요청 ("백엔드만", "QA만 다시", "이 부분 개선") | 부분 재실행 | 해당 에이전트만 재호출, 기존 산출물을 입력으로 전달 |
| `_workspace/` 있음 + 새 기능 요청 | 새 실행 | 기존 `_workspace/`를 `_workspace_prev/`로 이동 후 전체 파이프라인 |

추가 확인: `docs/contracts/` 존재 여부(기존 계약 재사용), 현재 코드 진척(`app/lib/`, `backend/src/`).

## Phase 1: 기획 (planner)

planner를 스폰해 요청을 분해시킨다. 산출: `_workspace/01_planner_plan.md`.

- 범위 판정이 "범위 외"면 여기서 중단하고 사용자에게 보고.
- "열린 질문"이 있으면 다음 Phase 진행 전 사용자에게 확인.

## Phase 2: 도메인 설계 (domain-analyst)

domain-analyst를 스폰해 설계·계약을 확정시킨다. 산출: `_workspace/02_analyst_design.md` + `docs/contracts/*.md`.

- 앱↔서버 통신이 포함된 작업이면 **계약 파일이 나오기 전에는 Phase 3을 시작하지 않는다** — 계약 없이 양쪽이 병렬 구현하면 경계면 버그가 보장된다.
- 순수 UI 작업(서버 무관)이면 이 Phase는 설계 검토만 하고 계약은 생략 가능.

## Phase 3: 구현 (backend-dev ∥ flutter-dev, 팀 모드)

계약 확정 후 backend-dev와 flutter-dev를 **병렬** 투입한다. 팀 모드에서는 둘이 SendMessage로 직접 API 질의를 주고받게 한다.

- 작업 의존성: 서버가 선행이어야 하는 항목(앱이 실서버 응답으로 검증해야 하는 경우)은 TaskCreate 의존 관계로 명시.
- 각 개발자는 모듈 완성 시 qa에게 통지 → **Phase 4를 기다리지 말고 점진 QA가 병행**된다.
- 순수 함수가 만들어지면 test-engineer에게 시그니처를 통지시킨다.

## Phase 4: 테스트 + QA (test-engineer ∥ qa)

- test-engineer: 통지받은 순수 함수의 골든 테스트 작성 (`golden-testing` 스킬 기준). 산출: `_workspace/05_test_report.md`.
- qa: 최종 통합 검증 — 점진 QA에서 남은 경계면 + 회귀 재검증. 산출: `_workspace/06_qa_report.md`.
- qa는 반드시 `general-purpose` 타입으로 스폰 (검증 스크립트 실행 필요).

## Phase 5: 종합 및 수렴

1. QA 차단급 발견 → 해당 개발자 재투입 → qa 재검증. 최대 2회 반복, 그래도 남으면 사용자에게 보고.
2. **회귀 방지 게이트 (필수):** 이번 실행에서 발견된 모든 버그(QA 차단·경고, 테스트 실패로 드러난 구현 버그, 사용자 신고)에 대해:
   - `docs/regressions.md`에 등록됐는지 확인 — 누락분은 지금 등록
   - 각 항목에 재발 방지 장치(재현 테스트 or QA 상시 점검 항목 or 절차 교정)가 연결됐는지 확인 — 없으면 test-engineer(테스트) 또는 qa(점검 항목)에게 박제 작업을 지시한 후에만 종결
   - 수정이 들어간 영역의 **기존 테스트 스위트 전체 재실행** — 이번 수정이 과거 버그를 되살리지 않았는지 확인
3. 전체 산출물 요약을 사용자에게 보고: 구현 내역, 테스트 결과, QA 판정, 회귀 레지스트리 갱신 내역, 미해결 항목, 다음 단계 제안.
4. 실행 후 피드백 기회 제공: "결과나 팀 구성에서 개선할 부분이 있나요?" (강요하지 않는다.)

## 데이터 전달 프로토콜

- **파일 기반 (산출물)**: `_workspace/{NN}_{agent}_{artifact}.md`. 중간 산출물은 보존(감사 추적). 계약은 예외적으로 `docs/contracts/`에 영구 보관.
- **태스크 기반 (조율)**: TaskCreate로 작업·의존 관계 관리.
- **메시지 기반 (실시간)**: 팀원 간 SendMessage — 계약 질의, 완성 통지, 차단급 발견 통지.

## 서브 에이전트 폴백 (팀 도구 부재 시)

Agent 도구로 순차/병렬 직접 호출한다. Phase 3의 팀원 간 직접 통신을 오케스트레이터가 중계한다: backend-dev 반환값에서 "flutter-dev에게 전달할 사항"을 추출해 flutter-dev 프롬프트에 포함. 병렬 실행은 한 메시지에 여러 Agent 호출로.

## 에러 핸들링

| 상황 | 대응 |
|------|------|
| 에이전트 실패/무응답 | 1회 재시도. 재실패 시 해당 산출물 없이 진행하되 최종 보고에 누락 명시 |
| 빌드·테스트 실패 지속 | 통과한 척 금지. 실패 로그와 함께 사용자 보고 |
| 계약 상충 (서버 vs 앱 주장 불일치) | 삭제·임의 판정 금지. 양쪽 주장 병기해 domain-analyst 판정 → 그래도 미해결이면 사용자 |
| 범위 침범 (2차/범위외 구현 시도) | planner의 범위 판정 기준으로 차단, 사용자에게 확인 |
| QA 차단 2회 반복 미해결 | 루프 중단, 현황 그대로 사용자 보고 |

## 테스트 시나리오

**정상 흐름 — "크루 생성/초대 기능 만들어줘":**
Phase 0(초기 실행 판별) → planner(MVP 판정, 작업 분해: 서버 Crew 컨텍스트 + 앱 크루 화면) → domain-analyst(Crew 불변식 체크리스트 + `docs/contracts/crew-api.md`) → backend-dev ∥ flutter-dev(계약 기반 병렬 구현, 완성 시 qa 통지) → test-engineer(InviteCode 만료·사용횟수 로직 테스트) ∥ qa(3자 대조: 계약 vs CrewController vs 앱 DTO) → 종합 보고.

**에러 흐름 — QA가 enum 불일치 발견:**
qa가 서버 `LEADER/MEMBER` vs 앱 `leader/member` 불일치를 차단급 보고 → 계약 확인(대문자가 기준) → flutter-dev 재투입(파서 수정) → qa 회귀 재검증 → 통과 → 종합 보고에 수정 이력 포함.

**에러 흐름 — 범위 침범:**
"실시간으로 크루원 위치 보이게 해줘" → planner가 2차 항목(실시간 위치 공유)으로 판정 → 구현하지 않고 사용자에게 "MVP 검증 후 2차 예정, 지금 진행하면 MVP 일정 지연" 트레이드오프 보고 → 사용자 결정 대기.

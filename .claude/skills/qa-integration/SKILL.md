# 통합 정합성 검증 (QA)

핵심 신조: **존재 확인은 QA가 아니다.** "엔드포인트가 있다, DTO가 있다"는 검증이 아니라 목록 작성이다. QA는 경계 양쪽 파일을 **동시에 열어 shape을 필드 단위로 대조**하는 행위다.

## 3자 대조 절차

경계면 하나당:

1. `docs/contracts/{이름}.md` 를 연다 — 이것이 기준.
2. 서버 쪽을 연다: 컨트롤러 + 요청/응답 record (backend/.../adapter/in/web).
3. 클라이언트 쪽을 연다: DTO/파서 (app/lib/core/model).
4. **필드 단위 표**로 대조: 이름(케이스 포함), 타입, null 허용, enum 값 집합, 단위, 포맷.
5. 가능하면 실행 대조: 서버 기동 → curl 실응답 캡처 → 계약과 diff. 불가하면 "정적 검증만"을 보고서에 명시.

계약이 없는 경계면을 발견하면 그 자체가 차단급 발견이다 — domain-analyst에게 소급 작성 요청.

## 전형적 경계면 버그 패턴 (우선 수색 목록)

| 패턴 | 이 프로젝트의 구체 지점 |
|------|------------------------|
| 필드명 케이스 (snake/camel) | 계약은 snake_case — Jackson/json_serializable 설정 양쪽 확인 |
| enum 문자열 불일치 | Participation: `DNF/DNS/...`, RaceStatus, RewardGrant 상태 — 서버 `@Enumerated(STRING)` 값과 앱 파서의 문자열 집합 비교 |
| 단위 불일치 | distance_m(미터) vs km 표시, record_time_s(초) vs ms — 필드명 접미사와 실값 모두 확인 |
| 시각 포맷·타임존 | Instant(UTC ISO-8601) vs 앱 DateTime 파싱, GPS 시각 vs 기기 시각 |
| 폴리라인 정밀도 | 인코딩 precision 1e5 vs 1e6 — 양쪽 라이브러리 기본값이 다르면 좌표가 10배 틀어진다 |
| null/옵셔널 | replay_notified_at, finished_at(DNF), withdrawn_at 등 nullable 필드의 앱 쪽 처리 |
| 병렬 배열 길이 | 업로드 페이로드의 폴리라인 포인트 수 == timestamp 배열 길이 == speed 배열 길이 |
| 페이지네이션·래핑 | 응답이 `{data: [...]}` 인지 bare 배열인지 |

## 상태머신·불변식 검증

`_workspace/02_analyst_design.md`의 불변식 체크리스트를 받아 코드에서 확인한다. 상시 점검 항목:

- **서버 RaceStatus ↔ 클라이언트 로컬 상태 대응**: FINISHED_LOCAL이 서버로 전송되는 코드가 있으면 버그 (서버가 모르는 상태).
- **코스 불변**: OPEN 이후 세션이 참조한 코스의 UPDATE 경로가 존재하는지 수색 — 있으면 차단급.
- **탈퇴 처리 구분**: 파기 대상(닉네임, kakao_id, 토큰, track_payload)과 익명 보존 대상(rank_entry, 스냅샷 내 경로)이 뒤바뀌지 않았는지.
- **알림 멱등**: replay_notified_at 확인 없이 FCM 발송하는 경로 수색.
- **track_payload 격리**: 순위·히스토리 조회 경로에서 payload 로드 여부 (JPA 연관, JOIN 쿼리 수색).
- **로컬 우선**: 업로드 성공 확인 전 로컬 삭제 코드 수색.
- **취소 정책**: CANCELLED 세션에서 순위·보상 생성 경로가 막혔는지 + 트랙 개인 기록 보존 여부.
- **릴리즈 매니페스트 스모크** (R-004): main AndroidManifest에 INTERNET 등 릴리즈 필수 권한이 있는지 — debug 오버레이가 가려주는 권한은 릴리즈에서만 터진다. 네트워크·권한 관련 기능 추가 시 debug/main 매니페스트를 쌍으로 확인.

## 점진적 QA 절차

모듈 완성 통지 수신 → 즉시 해당 경계면만 검증:

1. 이번 모듈이 만든/변경한 경계면 식별 (계약 diff, 신규 엔드포인트, 신규 DTO)
2. 3자 대조 + 해당 모듈 관련 불변식 확인
3. 실행 검증 (빌드, 테스트 스위트, 가능하면 서버 기동 + curl)
4. 보고: 차단(경계면 불일치·불변식 위반 — 즉시 해당 개발자 통지) / 경고(잠재 위험) / 참고

이전 보고서(`_workspace/06_qa_report.md`)가 있으면 **이전 차단·경고 항목의 수정 여부를 먼저 재검증**한 후 새 범위로 넘어간다 — 회귀 확인 없는 QA는 절반짜리다.

## 회귀 레지스트리 연동 (`docs/regressions.md`)

- **발견 즉시 등록**: 차단·경고급 발견은 보고서와 동시에 레지스트리에 등록한다 (ID 발번, 증상·원인).
- **종결은 박제 후에만**: "수정 확인"과 "재발 방지 장치 존재 확인"은 별개 단계다. 수정됐어도 재현 테스트나 상시 점검 항목이 없으면 OPEN으로 유지하고 오케스트레이터에 박제 미완을 보고한다.
- **상시 점검 항목 승격**: 테스트로 잡기 어려운 유형(설정, 컨벤션, 절차)은 이 스킬의 "상시 점검 항목" 목록에 추가하는 것이 박제다 — 항목을 추가하면 레지스트리에 그 사실을 기록한다.
- **매 QA 실행 시 레지스트리의 OPEN 항목을 훑고**, 이번 검증 범위와 겹치는 항목은 재발 여부를 다시 확인한다.

## 보고 형식

발견 하나당: 심각도 / 경계면 이름 / 양쪽 인용(`파일:라인`) / 재현 입력과 증상 / 계약 기준 어느 쪽이 틀렸는가 / 권고 수정 방향. "~인 것 같다"는 표현이 필요하면 그것은 발견이 아니라 추가 조사 항목이다 — 구분해서 적는다.

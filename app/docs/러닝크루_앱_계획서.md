# 러닝 크루 리플레이 앱 — 프로젝트 계획서

> 플랫폼: Flutter 단일 코드베이스 / **Android 우선 출시, iOS 후속 확장 전제**

## 1. 한 줄 정의

지인들끼리 초대 기반 크루를 만들어 정해진 코스를 달리고, 끝난 뒤 지도 위에서 모두의 이동 경로를 리플레이로 돌려보며 순위에 따라 보상(커피·음료·사탕)을 주고받는 클로즈드 러닝 게임 앱.

## 2. 핵심 가치와 차별점

순위표 숫자가 아니라 "어떻게 움직였는지"를 보는 재미가 제품의 중심이다. 레이스가 끝나면 참가자 전원의 마커가 지도 위에서 동시에 움직이는 고스트 레이스 리플레이가 생성되고, 추월 지점·구간 페이스가 시각화된다. 신뢰 기반 지인 그룹만을 대상으로 하므로 안티치트를 배제하고, 그 리소스를 리플레이 경험과 게임성(핸디캡, 개인 기록 갱신, 벌칙)에 집중한다.

## 3. 범위 정의

### MVP (1차 목표)

- 초대 코드 기반 크루 생성·참가 (공개 모집 없음)
- 코스 등록: 지도에서 경로 그리기 또는 과거 주행 기록을 코스로 승격
- 레이스 세션 생성: 코스, 일시, 참가자, 보상 내용(텍스트) 지정
- GPS 트래킹: 3~5초 간격 백그라운드 위치 기록, 완주 후 서버 업로드
- 결과 처리: 전원 업로드 완료 시 순위 확정, 리플레이 생성
- 리플레이 뷰어: 배속 재생, 시간축 슬라이더, 추월 지점 마킹, 구간 페이스 색상
- 순위표·기록 히스토리: 세션별 순위, 개인 누적 기록, 개인 최고 기록(PB)
- 보상 지급: 수동(카카오톡 선물하기 등), 앱은 "누가 누구에게 무엇을" 기록만 관리
- 푸시 알림(FCM): 세션 리마인더, 결과 확정("리플레이가 열렸어요") — 리플레이 조회율 목표 달성의 전제 기능

### 2차 (MVP 검증 후)

- 실시간 위치 공유: 달리는 중 참가자 위치 브로드캐스트 (WebSocket/SSE)
- 비동기 레이스: 기간 내 각자 원하는 시간에 코스를 뛰고 기록 경쟁 — 상대 시각 동기화 리플레이 구조상 추가 비용 낮음, 일정 안 맞는 주의 리텐션 보완
- 핸디캡 시스템: 과거 기록 기반 출발 시차 또는 보정 기록
- 벌칙/보너스 룰: 꼴등 벌칙, PB 갱신 보너스 등 크루별 커스텀 룰
- 기프티콘 API 연동(기프티쇼 비즈 등)으로 보상 자동 발송

### 범위 외 (명시적 제외)

- 부정행위 탐지(GPS 스푸핑, 이동수단 판별) — 신뢰 기반 그룹 전제
- 공개 크루 검색·모집, 소셜 피드
- 참가비 징수 — 사행성 이슈 회피

## 4. 사용자 흐름

1. 크루장이 크루를 만들고 초대 링크/코드를 카톡방에 공유한다.
2. 멤버들이 코드로 참가한다.
3. 크루장이 코스를 그리고 레이스 세션을 만든다("토요일 오전 8시, 한강 5km, 1등 스타벅스").
4. 당일 각자 앱에서 '레이스 시작'을 누르고 뛴다. 폰은 주머니에 있어도 백그라운드에서 기록된다.
5. 완주 후 '종료'를 누르면 기록이 서버로 업로드된다.
6. 전원 업로드 완료(또는 마감 시각 도달) 시 순위가 확정되고 리플레이가 열린다.
7. 뒤풀이에서 리플레이를 돌려보고, 크루장이 보상 지급을 체크한다.

### 레이스 규칙 (MVP 확정)

- **기록 방식은 그로스 타임(gross time)**: 자동 일시정지 없음 — 신호등 대기도 기록에 포함된다. 단순하고 분쟁 여지가 없으며, 정지 감지 오작동 리스크를 제거한다. 네트 타임/자동 일시정지는 크루 피드백을 보고 2차에서 재검토.
- **기록 구간은 시작 버튼 ~ 도착점 최초 진입**: 기록 종료 시각은 도착점 반경 최초 진입 시각으로 **자동 확정**한다 — 종료 버튼을 늦게 눌러도 기록이 손해 보지 않게. 종료 버튼은 트래킹 중단과 업로드 트리거 역할만 한다(도착점 미진입 상태로 종료하면 DNF 후보).
- **출발은 각자 시작 버튼 기준**: 같은 장소 동시 출발을 강제하지 않는다. 순위는 개인 소요 시간으로 산정하므로 출발 시각이 달라도 공정하다.
- **리플레이는 상대 시각 동기화**: 각자의 시작 시점을 t=0으로 정렬해 병합한다. 이 구조 덕에 2차의 비동기 레이스(며칠에 걸쳐 각자 뛰기)가 리플레이 로직 변경 없이 지원된다.
- **완주 판정**: ① 도착점 반경 30m 이내 진입, ② 정제 후 주행 거리가 코스 거리의 90% 이상, ③ 코스 일치율 — 정제 트랙 포인트의 일정 비율(80%) 이상이 코스 폴리라인으로부터 허용 거리(50m) 이내. 세 조건 모두 충족 시 완주. 지름길 주파나 다른 길로 잘못 든 경우를 올바르게 판정하기 위함이며(안티치트 아님), 미충족 시 DNF 처리하되 기록·경로는 보존해 리플레이에는 표시한다. 임계값(30m/90%/80%/50m)은 운영하며 튜닝한다.

## 5. 도메인 모델

바운디드 컨텍스트 6개로 나눈다. 컨텍스트 간 결합은 이벤트로 풀고, MVP에서는 모놀리스 내 패키지 분리(헥사고날)로 시작한다.

### 5.0 User 컨텍스트 — 누구인가

| 구분 | 이름 | 설명 |
|---|---|---|
| Aggregate Root | **User** | 회원. 닉네임, 프로필, 상태(ACTIVE/WITHDRAWN) |
| Value Object | **KakaoAccount** | 카카오 회원번호(연동 키) — 인증 어댑터 외부로 노출하지 않음 |
| Value Object | **DeviceToken** | FCM 푸시 토큰. 기기 교체 시 갱신 |
| Domain Event | UserWithdrawn | 탈퇴 → 각 컨텍스트의 후처리 트리거 |

불변식 및 탈퇴 정책: 탈퇴 시 개인 식별 정보(닉네임, 카카오 연동 키, 푸시 토큰)는 즉시 파기하고, 크루 내 과거 레이스 기록·순위는 "탈퇴한 러너"로 **익명화 보존**한다 — 삭제하면 남은 크루원들의 리플레이·순위 히스토리가 깨지기 때문. 위치 원본(track_payload의 raw/refined)은 탈퇴 시 지체 없이 삭제한다. 이미 생성된 ReplaySnapshot 안의 경로는 식별자와 결합이 끊긴 **익명 트랙으로 보존**한다 — "식별 정보와 위치 원본은 파기, 식별자가 분리된 파생물은 익명 보존"이라는 구분을 개인정보 처리방침에 그대로 명시한다.

### 5.1 Crew 컨텍스트 — 누가 함께 뛰는가

| 구분 | 이름 | 설명 |
|---|---|---|
| Aggregate Root | **Crew** | 크루. 이름, 크루장, 멤버 목록, 상태 |
| Entity | **CrewMember** | 멤버십. 역할(LEADER/MEMBER), 가입일, 상태 |
| Value Object | **InviteCode** | 초대 코드. 만료 시각, 사용 횟수 제한 |
| Domain Event | CrewMemberJoined | 멤버 합류 → 알림 등 |

불변식: 크루장은 항상 1명 존재. 초대 코드로만 가입 가능(공개 가입 경로 없음). **크루장이 탈퇴하면 가입일이 가장 빠른 멤버에게 자동 승계**하고, 마지막 1인이 탈퇴하면 크루는 폐쇄(CLOSED) 처리한다 — 크루장 부재 상태가 존재하지 않도록.

### 5.2 Race 컨텍스트 — 언제 어디서 뛰는가

| 구분 | 이름 | 설명 |
|---|---|---|
| Aggregate Root | **RaceSession** | 레이스 세션. 코스, 예정 일시, 상태, 업로드 마감 시각 |
| Aggregate Root | **Course** | 코스. 경로 폴리라인, 총 거리, 출발/도착 지점, 생성자 |
| Entity | **Participation** | 참가. 참가자, 서버 측 상태(REGISTERED/STARTED/FINISHED/DNF/DNS/WITHDRAWN — DNS는 신청 후 미출주) |
| Value Object | **RoutePath** | 인코딩된 경로 좌표열 + 거리 |
| Value Object | **RaceStatus** | DRAFT → OPEN → RUNNING → FINALIZING → COMPLETED / CANCELLED |
| Domain Service | **FinishPolicy** | 완주 판정: 도착점 반경 진입 + 주행 거리 + 코스 일치율(레이스 규칙 참조). 코스 이탈 검증은 여기로 일원화. 순수 함수로 구현해 골든 테스트 대상 |
| Domain Event | RaceCompleted | 전원 업로드 또는 마감 도달 → Ranking·Replay 트리거 |

불변식: **발행(OPEN 이후 세션에서 사용)된 코스는 불변** — 수정이 필요하면 새 코스를 생성한다. 코스를 고치면 그 코스를 참조하는 과거 리플레이·PB의 의미가 깨지기 때문. COMPLETED는 전원 업로드 완료 또는 마감 시각 경과 시에만 진입.

취소 정책: 크루장은 **RUNNING 중에도 세션을 취소할 수 있다**(우천·사고 등 현실 대응). CANCELLED 세션은 순위·보상을 생성하지 않되, 이미 뛰고 있던 참가자의 트랙은 업로드 시 **개인 기록(세션 무관 주행)으로 보존**한다 — 뛴 노력이 증발하지 않게.

참고 — 클라이언트 로컬 상태머신은 별도로 둔다: READY → RUNNING → FINISHED_LOCAL → UPLOADED. "완주했지만 서버가 죽어 업로드 대기 중"(FINISHED_LOCAL)은 서버가 알 수 없는 상태이므로 서버 상태에 두지 않는다 — 로컬 우선 저장 원칙의 클라이언트 측 표현. '레이스 시작' 시 서버에 STARTED 신호를 한 번 보내 크루원들이 "지금 뛰는 중"인 사람을 볼 수 있게 한다(서버 다운 시 이 신호는 생략 가능, 기록에 영향 없음).

### 5.3 Tracking 컨텍스트 — 어떻게 움직였는가 (데이터 심장부)

| 구분 | 이름 | 설명 |
|---|---|---|
| Aggregate Root | **TrackRecord** | 한 참가자의 한 레이스 주행 기록. 시작/종료 시각, 총 거리, 총 시간 |
| Value Object | **TrackPoint** | (timestamp, lat, lng, altitude, speed, accuracy) 단일 샘플 |
| Value Object | **TrackSegment** | 구간(예: 500m 단위) 페이스 요약 — 리플레이 색상용 |
| Domain Service | **TrackRefinementService** | GPS 정제: accuracy 임계 초과 포인트 제거, 이동 평균 스무딩, 이상 점프 보정. 총 거리·페이스는 **정제 후 좌표로 계산** — 원시 좌표 하버사인 누적은 노이즈로 거리가 부풀기 때문 |
| Domain Service | **TrackUploadService** | 업로드 수신, 폴리라인 디코딩, 기본 검증(코스 이탈 허용 오차, 시간 정합성) |
| Domain Event | TrackUploaded | 업로드 완료 → Race 컨텍스트에 통지 |

설계 노트: TrackPoint 샘플링은 **적응형** — 주행 중 3~5초, 속도 0(정지 감지) 지속 시 10초로 완화, 이동 재개 시 복귀. 배터리 소모를 유의미하게 줄인다. 전송은 인코딩 폴리라인 + 병렬 배열(timestamp/speed). 타임스탬프는 기기 시계가 아닌 **GPS 시각 우선** 사용 — 기기 시계 오차로 인한 기록·동기화 왜곡 방지. 원시 포인트와 정제 결과를 모두 보존해(원시는 재정제 가능하도록) 정제 알고리즘을 개선해도 과거 기록을 다시 계산할 수 있게 한다.

저장은 **track_record(메타·요약)와 track_payload(raw/refined 블롭)를 1:1 테이블로 분리**한다 — TrackRecord 애그리거트는 하나로 유지하되 저장 구조만 나누는 것. 순위·PB·히스토리 조회는 track_record만 스캔하므로 수백 KB 블롭이 엔티티 로드에 딸려오는 실수와 버퍼 풀 낭비를 원천 차단하고, payload 접근은 리플레이 생성·재정제 시 전용 리포지토리 경로로 한정한다. 트랙은 업로드 시점에 한 번 쓰이고 불변이므로 트랜잭션 경계 문제는 없다.

### 5.4 Ranking 컨텍스트 — 누가 이겼는가

| 구분 | 이름 | 설명 |
|---|---|---|
| Aggregate Root | **RaceResult** | 세션 결과. 확정된 순위 목록, 확정 시각 |
| Entity | **RankEntry** | 개인 결과. 순위, 기록, 평균 페이스, PB 여부 |
| Value Object | **PersonalBest** | 유저×코스(course_id) 기준 최고 기록 — **코스가 불변이므로 PB의 참조 대상이 흔들리지 않는다**. 역방향·변형 코스는 새 코스이므로 자연히 별도 PB. 거리대(5km/10km) 기준 PB는 2차 |
| Value Object | **Handicap** (2차) | 보정 규칙. 출발 시차형 / 기록 보정형 |
| Domain Service | **RankingPolicy** | 순위 산정: 완주 기록 오름차순. **동률은 공동 순위 + 다음 순위 건너뜀**(1, 1, 3위). DNF/DNS는 순위 미부여로 목록 하단 표기 |
| Domain Event | ResultFinalized | 순위 확정 → Reward 컨텍스트 트리거 |

설계 노트: 실시간 랭킹이 필요해지는 2차부터 Redis Sorted Set 도입. MVP는 업로드 완료 시점 일괄 산정이라 RDB만으로 충분.

### 5.5 Reward 컨텍스트 — 무엇을 주고받는가

| 구분 | 이름 | 설명 |
|---|---|---|
| Aggregate Root | **RewardPlan** | 세션의 보상 계획. "1등: 커피, 2등: 음료, 3등: 사탕" |
| Entity | **RewardGrant** | 지급 건. 수령자, 내용, 상태(PENDING/SENT/CONFIRMED) |
| Value Object | **RewardItem** | 보상 항목. MVP는 자유 텍스트 + 예상 금액 |
| Domain Event | RewardGranted | 지급 확정 → 알림 |

설계 노트: MVP는 수동 발송이므로 상태 관리 장부 역할만 한다. 2차에서 기프티콘 API 연동 시 RewardGrant에 발송 어댑터를 붙이고, 발송 실패 재시도는 Outbox 패턴으로 처리 — 지급 확정 트랜잭션과 외부 발송을 분리해야 중복 발송/유실을 막는다.

### 5.6 Replay — 프로젝션 계층 (별도 패키지)

리플레이는 별도 애그리거트를 두지 않는 **프로젝션 계층**이며, Tracking에도 Ranking에도 속하지 않는 **독립 패키지 `replay`** 로 둔다: `ReplayGenerationService`(트랙 → 스냅샷 계산), `ReplaySnapshotRepository`(저장·조회). 개념상 파생 뷰라는 정의와 코드 배치를 일치시키는 것.

세션 참가자 전원의 트랙을 **각자의 시작 시점 t=0 기준 상대 시각으로 정렬·병합**한 **ReplaySnapshot**으로 저장한다. 스냅샷은 `schema_version`과 **생성 상태(GENERATING → READY / FAILED)** 를 가진다 — 비동기 생성이므로 실패가 관측 가능해야 하고, FAILED는 "리플레이가 안 열려요"를 조용한 부재가 아닌 재시도 가능한 상태로 만든다. 스키마 버저닝 덕에 추월 표시, 고스트, 페이스 색상 등 표현이 진화해도 과거 레이스(이 제품의 핵심 자산인 추억 아카이브)가 깨지지 않으며, 원시 트랙이 보존되는 한 **스냅샷은 언제든 삭제 후 최신 스키마로 재생성 가능(멱등)** 하다 — 이 삭제→재생성은 운영 도구(관리 API 또는 스크립트)로 처음부터 마련한다. 운영하다 보면 반드시 쓰게 되는 기능.

생성은 **ResultFinalized 커밋 이후 비동기**로 수행한다 — 순위 확정 트랜잭션과 스냅샷 계산을 분리해 확정이 계산 시간에 인질 잡히지 않게 한다. MVP 규모(10명 × 600포인트)에선 수백 ms 수준이라 잡 큐 없이 AFTER_COMMIT 이벤트 리스너로 충분하다. 최초 READY 도달 시점이 FCM "리플레이 열림" 알림의 트리거이며, **알림은 세션당 1회 멱등** — 재생성으로 READY가 다시 발생해도 재발송하지 않는다(발송 여부를 세션에 기록). 클라이언트는 스냅샷 하나로 마커 보간 애니메이션, 추월 지점, 구간 페이스 색상을 렌더링하며, 추월 지점 계산(동일 진행거리 도달 시각 비교)은 서버에서 사전 계산해 포함시킨다. 추월·순위 산정과 함께 순수 함수로 구현해 골든 테스트(입력 트랙 → 기대 결과 고정)를 깔아둔다.

### 컨텍스트 간 이벤트 흐름

```
TrackUploaded ─▶ TrackRefinement(정제) ─▶ FinishPolicy(완주/DNF 판정)
             ─▶ RaceSession(전원 완료 체크) ─▶ RaceCompleted
RaceCompleted ─▶ RankingPolicy 산정 ─▶ ResultFinalized
ResultFinalized ─(커밋 후 비동기)▶ ReplaySnapshot GENERATING → READY ─▶ FCM "리플레이 열림"(세션당 1회)
              │                                        └─▶ FAILED → 운영 재생성
              └▶ RewardGrant 생성 (PENDING)
```

## 6. 기술 아키텍처

**플랫폼 전략**: Flutter 단일 코드베이스, **Android 우선 출시 → iOS 후속 확장**. Android 전용 구현이 앱 전반에 스며들지 않도록, 플랫폼 의존이 발생하는 지점을 처음부터 격리한다.

- **트래킹 계층 추상화**: `LocationTracker` 인터페이스(start/stop/pause, 포인트 스트림, 상태)를 정의하고 MVP는 `AndroidForegroundTracker` 구현만 제공한다. iOS 확장 시 `IosBackgroundTracker`(Always 권한 + 위치 백그라운드 모드)를 같은 인터페이스로 추가하며, 상위 레이어(레이스 화면, 업로드 파이프라인)는 수정 없이 유지된다.
- **OS 종속 지점의 추가 추상화**: `NotificationService`(포그라운드 서비스 알림 vs iOS 로컬 알림), `PermissionService`(위치·알림 권한 요청 플로우가 OS별로 상이)도 인터페이스로 격리한다 — 트래커와 함께 iOS 확장 시 구현체만 추가하는 3개 지점.
- **플랫폼 무관 코어**: 트래킹 버퍼링·로컬 저장·업로드 재시도·폴리라인 인코딩은 순수 Dart로 작성해 양 플랫폼 공용. 권한 요청·배터리 안내 등 OS 종속 UX만 플랫폼 분기.
- **의존 패키지 선정 기준**: 양 플랫폼을 모두 지원하는 패키지 우선 — `geolocator`(iOS/Android 공용) + `flutter_foreground_task`(Android 서비스 계층), `flutter_naver_map`(iOS 지원 포함), `kakao_flutter_sdk`(공식, 양 플랫폼). Android 전용 패키지는 트래커 구현체 내부로만 한정한다.
- **서버는 플랫폼 무지**: 업로드 스키마·API에 플랫폼 종속 필드를 두지 않는다. 디버깅용 `client_meta`(os, os_version, device_model)만 부가 정보로 수집.

**클라이언트(Android 1차)**: 백그라운드 위치는 **Foreground Service(type: location) + 상시 기록 알림** 방식 — `ACCESS_BACKGROUND_LOCATION` 선언 없이 화면 꺼짐 상태 트래킹이 가능하고 Play의 백그라운드 위치 사용 심사를 우회한다. 제조사 절전 정책(특히 삼성 앱 자동 종료) 대응으로 온보딩에 배터리 최적화 예외 안내를 포함한다. 지도는 네이버 지도 SDK(국내 보행로 품질, iOS 동시 지원).

**iOS 확장 시 추가 작업(예약)**: Always 권한 온보딩 플로우, 위치 백그라운드 모드 설정, TestFlight 배포 채널, 카카오 로그인 iOS 설정. 트래커 구현체 1개 추가 외 코어 변경 없음이 목표.

**백엔드**: Spring Boot 3.x + Java 17, 헥사고날 구조(컨텍스트별 패키지 분리). MVP는 REST만으로 충분 — 트래킹 업로드, 세션 CRUD, 리플레이 JSON 조회. 2차 실시간 공유 도입 시 WebSocket(STOMP) 또는 SSE 추가.

**저장소**: MySQL 단일. 트랙은 track_record(메타·요약)와 track_payload(블롭) 1:1 분리 저장(5.3 설계 노트 참조). 리플레이는 **replay_snapshot 테이블에 스키마 버전·생성 상태와 함께 저장** — 재생성 불필요해지고, 일일 DB 백업 흐름에 리플레이가 자동 포함된다. Redis는 2차(실시간 랭킹·위치 브로드캐스트 pub/sub)부터.

**인증**: 카카오 로그인 단일 — 대상이 전원 카톡 사용자이고 초대 링크도 카톡으로 도니 마찰 최소.

**알림**: FCM. MVP는 두 종류만 — 세션 리마인더(예정 시각 전), 결과 확정(리플레이 오픈). DeviceToken은 User 컨텍스트에서 관리.

**인프라 — 로컬 서버 + Cloudflare Tunnel**: 서버는 로컬 컴퓨터(홈서버)에서 Docker Compose(Spring Boot + MySQL)로 구동하고, **Cloudflare Tunnel(cloudflared)** 로 외부에 노출한다. 포트포워딩 없이 아웃바운드 연결만으로 고정 도메인 + HTTPS가 확보되고, 공인 IP·가정용 회선 정보가 은닉되며 비용이 0원이다. 도메인은 Cloudflare에 등록된 것 하나면 된다(api.<도메인> 서브도메인 권장).

로컬 서버의 구조적 한계는 **가용성**이다 — 정전, 재부팅, 회선 장애 시 서버가 내려간다. 이를 아키텍처로 완충한다:

- **클라이언트는 서버 없이도 레이스가 가능해야 한다**: 트래킹·기록은 전부 로컬 우선 저장이고, 업로드는 사후 비동기 + 지수 백오프 재시도. 레이스 도중 서버가 죽어도 기록은 유실되지 않고, 서버 복구 후 업로드되면 순위·리플레이가 정상 생성된다. 이 원칙은 이미 트래킹 설계에 있으므로 "서버 다운 = 결과 지연"일 뿐 "결과 유실"이 아니게 유지한다.
- **백업**: MySQL 일일 덤프를 만들어 로컬 외 위치(클라우드 스토리지 또는 별도 기기)에 보관. 홈서버 디스크 단일 장애점을 제거한다.
- **재기동 자동화**: Docker 컨테이너 restart 정책 + cloudflared 서비스 등록으로 정전 복구 시 무인 복원.
- **공개 전환 시 이전 경로**: 트래픽·가용성 요구가 커지면 컨테이너 그대로 클라우드로 이전한다(후보: Oracle Cloud Free Tier, AWS Lightsail, Hetzner 등 저비용 VPS). Cloudflare 도메인 뒤에 있으므로 클라이언트 수정 없이 원점만 교체하면 된다.

**운영·관측**: 앱은 Firebase Crashlytics(크래시)와 업로드 실패율 로깅, 서버는 구조화 로그 + 디스크 사용량·헬스체크(Cloudflare 헬스 알림 또는 간단한 외부 uptime 모니터). 그리고 **최소 지원 버전 API**(`GET /app-version` → 강제 업데이트 판단)를 초기부터 넣는다 — 초기엔 업로드 스키마가 자주 바뀌므로 구버전 클라이언트를 차단할 수단이 필수다.

**테스트 전략**: 완주 판정(FinishPolicy), GPS 정제(TrackRefinement), 추월 계산·순위 산정(RankingPolicy, ReplaySnapshot)은 전부 순수 함수로 격리해 골든 테스트를 우선 배치한다. 실제 주행에서 수집한 원시 트랙을 픽스처로 축적하면 정제 알고리즘 튜닝의 회귀 방지선이 된다. 도메인 계층 TDD, 어댑터(업로드 API, FCM)는 통합 테스트 최소한으로.

## 8. 마일스톤

**M0 — 스토어·인프라 준비 (즉시)**: Play Console 계정 생성, 패키지명(applicationId) 확정, 릴리즈 서명 키 생성 후 Android 개발자 인증(패키지명 + SHA-256 등록, 2026년 4월부터 필수) 완료. 카카오 개발자 콘솔 앱 생성 — 테스트 단계에서는 **팀원으로 등록된 카카오 계정만 로그인 가능**하므로 크루원 계정을 팀원 등록하고, 프로덕션 전환 전 동의항목 검수 요건을 확인한다. 네이버 클라우드 플랫폼 콘솔에 지도 SDK 앱 등록, Firebase 프로젝트 생성(FCM·Crashlytics), 도메인의 Cloudflare 등록 + Tunnel 구성(홈서버 → api 서브도메인). **개인정보처리방침·이용약관 초안 작성** — Play 등록 시 방침 URL과 데이터 보안 양식이 필수이며 위치정보 수집 항목이므로 수집 목적·보관 기간·탈퇴 시 파기 정책(User 컨텍스트 참조)을 명시한다. 방침 페이지는 Cloudflare Pages로 무료 호스팅.

**M1 — 트래킹 스파이크 + 뼈대 (2~3주)**: **첫 주는 기술 스파이크** — 로그인·화면 없이 빈 앱 + Foreground Service만으로 화면 끈 채 1시간 이상 백그라운드 기록, 실주행 1회를 갤럭시 실기기에서 검증한다. 이 프로젝트의 성립 조건이므로 다른 어떤 기능보다 먼저 확인하고, 유실이 심하면 이 시점에 유료 패키지(`flutter_background_geolocation`) 전환을 결정한다(fail fast). 스파이크 통과 후 카카오 로그인(Android 키 해시 등록 포함) + User 컨텍스트, 크루/초대, 코스 그리기, 세션 생성, 최소 지원 버전 체크를 진행하며, `LocationTracker`·`NotificationService`·`PermissionService` 인터페이스와 플랫폼 무관 코어(버퍼링·로컬 저장·업로드) 구조를 이 단계에서 확정한다.

**M2 — 트래킹 완성 (2~3주)**: `AndroidForegroundTracker` 구현(Foreground Service), GPS 정제(TrackRefinement) + 완주 판정(FinishPolicy) 골든 테스트, 업로드 파이프라인, 순위 확정. 이 구간이 리스크 최대 — 갤럭시 중심 실기기 테스트(절전 모드, 앱 자동 종료, 화면 꺼짐 장시간)를 넉넉히 잡고, 실주행 원시 트랙을 테스트 픽스처로 수집한다.

**M3 — 리플레이·알림 (2주)**: ReplaySnapshot 생성 로직(상대 시각 동기화, 스키마 버저닝, 커밋 후 비동기 생성), 뷰어(배속·슬라이더·추월 마킹·페이스 색상), FCM 알림 2종(리마인더·리플레이 열림). 제품의 얼굴이므로 뷰어 폴리싱에 투자.

**M4 — 비공개 테스트 = 실전 운영 (4주)**: Play 비공개 테스트 트랙으로 지인 배포. 개인 개발자 계정의 프로덕션 승인 요건(테스터 12명 이상, 14일 연속 참여)을 이 기간에 함께 충족한다 — 크루원 외 설치 유지용 예비 테스터를 여유 있게 확보. 주 1회 레이스를 4주 운영하며(성공 기준 측정 기간과 일치) 피드백 수집, 보상은 수동 지급. 카카오 로그인은 이 단계까지 팀원 등록 방식으로 운영한다.

**M5 — 게임성·자동화 (검증 후)**: 핸디캡, 벌칙 룰, 실시간 위치 공유, 기프티콘 API. 공개 전환 시점에 LBS 사업자 신고 진행.

**M6 — iOS 확장 (수요 확인 후)**: `IosBackgroundTracker` 구현, Always 권한 온보딩, TestFlight 배포. 코어 재사용을 전제로 목표 기간 2~3주.

## 9. 리스크와 대응

| 리스크 | 영향 | 대응 |
|---|---|---|
| 백그라운드 GPS 기기별 불안정 | 기록 유실 → 신뢰 붕괴 | 로컬 우선 저장 후 업로드, 유실 구간 보간 표시, 실기기 다종 테스트 |
| 홈서버 다운(정전·재부팅·회선 장애) | 업로드·조회 불가 | 클라이언트 로컬 우선 + 재시도로 "지연은 있어도 유실은 없게", 컨테이너 자동 재기동, 일일 DB 외부 백업, uptime 모니터 |
| 제조사 절전 정책의 서비스 강제 종료(삼성 등) | 트래킹 중단 | Foreground Service + 상시 알림, 온보딩에서 배터리 최적화 예외 안내, 갤럭시 실기기 검증 |
| Play 비공개 테스트 인원 이탈(12명 미달) | 프로덕션 승인 거절 → 재시작 | 크루원 외 예비 테스터 확보, 14일간 설치 유지 안내 |
| 전원 업로드 지연으로 결과 미확정 | 리플레이 못 봄 | upload_deadline 자동 확정 + 미업로드자 DNF 처리 |
| 배터리 소모 불만 | 사용 기피 | 3~5초 적응형 샘플링, 정지 감지 시 주기 완화 |
| Android 전용 구현의 코어 침투 | iOS 확장 비용 급증 | LocationTracker 인터페이스 격리, 순수 Dart 코어 유지, 양 플랫폼 지원 패키지 우선 |
| 공개 전환 시 LBS 미신고 | 법적 리스크 | 비공개 테스트 트랙 유지, 공개 전 신고 완료를 게이트로 설정 |

## 10. MVP 성공 기준

지인 크루 1개(6~10명)가 4주간 주 1회 레이스를 자발적으로 지속하고, 레이스 후 24시간 내 리플레이 조회율이 참가자의 80% 이상이면 리플레이 가설이 검증된 것으로 본다. 이후 실시간 공유·게임성 투자 여부를 결정한다.
